1. THE PRODUCT THE TEAM IS ACTUALLY BUILDING
=============================================

Storyverse is a collaborative visual-storytelling platform for a small comic or animated-short
team. The MVP must demonstrate one reliable journey:

1. Sign in and open a story project.
2. Upload or create a character.
3. Refine and lock the character reference.
4. Create a short sequence of visual scenes.
5. Generate or display one panel using project and canon context.
6. Create an alternate branch from a decision scene.
7. Simulate a collaborator changing that branch.
8. Compare the alternate branch with canon.
9. Run an AI continuity review that catches a specific contradiction.
10. Select changes and merge them into canon.
11. Show the updated branch tree and activity history.

The demo story should use the PRD's simple flooded-city scenario: an explorer, a companion or
stranger, a glowing compass, and one major choice. The contradiction can be that the explorer
uses the compass after it was lost or given away in an earlier scene.

DO NOT BUILD DURING THESE EIGHT DAYS:

- A complete comic editor or novel editor
- Full real-time Google Docs-style editing
- A marketplace or advanced licensing system
- A full 3D game-like interface
- Voice, audio, animation, or video export
- Public forking beyond a visual mock or future-work note
- Dozens of AI agents
- A full knowledge graph unless every must-have feature is already stable


2. OWNERSHIP MODEL
==================

Each feature has one directly responsible owner. Other teammates may review or assist, but the
owner is responsible for getting the feature integrated, tested, and demo-ready. Every person
should work on a separate feature branch and open small pull requests instead of one large PR.

Recommended branch names:

- feature/rahat-platform
- feature/omit-workspace
- feature/farin-ai
- feature/firdosi-character-review

Merge into the shared development branch at least once per day. Do not wait until Day 7 to
combine the four workstreams.

- Add timeout, malformed-response, rate-limit, and missing-credential handling.

  availability.
- Write tests for prompt/context assembly, schema validation, and the known compass conflict.
- Document which IBM model is used, what data is sent, and where human approval is required.

Main deliverables:

- Server-side watsonx/Granite integration
- Canon context builder
- Continuity inspector with a concrete, evidence-backed finding

8. DEFINITION OF DONE FOR EACH FEATURE
======================================

A feature is done only when:

- It works in the shared development deployment, not only on one laptop.
- It uses real persisted data or the documented deterministic demo fallback.
- Loading, empty, error, and success states are handled.
- Important behavior has a test.
- Keyboard focus and text contrast are usable.
- No secret or private data is sent to the client or committed to Git.
- AI output is labeled, explained, and requires human approval when it affects canon.
- The owner has updated relevant README or architecture notes.
- Another teammate has reviewed the pull request.
